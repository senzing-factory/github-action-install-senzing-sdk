## [5.0.1] - 2026-05-14

### Fixed in 5.0.1

- macOS staging `brew tap` failed to authenticate against the private tap. `GIT_ASKPASS` doesn't survive brew's subprocess env filter, so git fell back to interactive prompting and exited with `could not read Username for 'https://github.com'`. `tap-with-token` now uses a temporary `git config --global url.<auth>.insteadOf <public>` rewrite around the `brew tap` call instead â€” git reads `~/.gitconfig` from disk, which brew can't strip. The insteadOf entry is set immediately before `brew tap` and unset immediately after so the token's window in the config file is bounded by that single call.
- Windows pinned scoop install left the SDK at `~/scoop/apps/senzingsdk-pinned/current` (scoop derives the app name from the manifest filename, and the temp manifest was `senzingsdk-pinned.json`). `link-scoop-prefix` and `verify-installation` both look up `~/scoop/apps/senzingsdk/current`, so the install reported success but the post-install symlink failed. Renamed the temp manifest to `/tmp/senzingsdk.json` so scoop installs as `senzingsdk`, matching the rest of the script.

## [5.0.0] - 2026-05-14

### Changed in 5.0.0

- **Breaking:** root action now references the `@v5` internal subactions. Callers must pin to `@v5` (or `@main`) â€” `@v4` users keep the old DMG-only behavior on macOS.

### Added in 5.0.0

- `darwin-installer` input (macOS only): selects `homebrew` or `native`. Defaults to auto-detect (`homebrew` for pinned SDK â‰Ą 4.3.0 and for floating tags `staging-vN`/`production-vN`; `native` for pre-4.3.0 pinned versions). Requesting `homebrew` against a pre-4.3.0 pinned version warns and falls back to `native`.
- `senzingsdk-token` input (macOS and Windows): GitHub token used to clone the private staging Homebrew tap (macOS) or Scoop bucket (Windows). Defaults to `${{ github.token }}`.
- `windows-installer` input (Windows only): selects `scoop` or `native`. Defaults to auto-detect (`scoop` for pinned SDK â‰Ą 4.3.0 and for floating tags `staging-vN`/`production-vN`; `native` for pre-4.3.0 pinned versions). Requesting `scoop` against a pre-4.3.0 pinned version warns and falls back to `native`.
- macOS Homebrew install path: taps `Senzing/senzingsdk` (production) or `senzing-factory/senzingsdk-staging` (private) and runs `brew install --cask`. Pinned `X.Y.Z.BUILD` versions are forwarded via `HOMEBREW_SENZING_SDK_VERSION`; pinned `X.Y.Z` resolves the latest build from S3 before installing. `$HOME/senzing` is symlinked to the Homebrew install for backward compatibility. `DYLD_LIBRARY_PATH` is exported via `$GITHUB_ENV` with both the SDK's `er/lib` and Homebrew's `openssl@3/lib` so consumers don't have to wire OpenSSL up separately (libSz loads `@rpath/libssl.3.dylib` but its rpath doesn't include the Homebrew openssl@3 prefix).
- Windows Scoop install path: clones `Senzing/scoop-senzingsdk` (production) or `senzing-factory/scoop-senzingsdk-staging` (private) into `$HOME/scoop/buckets/senzingsdk` and runs `scoop install`. Pinned `X.Y.Z.BUILD` versions install from a temp manifest generated by the action (with the resolved SHA-256); pinned `X.Y.Z` resolves the latest build from S3 first. `$HOME/Senzing` is symlinked to the scoop install for backward compatibility. Both `SENZING_ACCEPT_EULA` (prod) and `SENZING_EULA_ACCEPTED` (staging) env vars are set during install to cover both buckets' EULA-acceptance conventions.

## [4.0.0] - 2026-03-12

### Changed in 4.0.0

- `senzingsdk-version` is now a required input
- `senzingsdk-repository-path` now requires `senzingsdk-version` to be set (previously defaulted to major version 4)
- Refactored install scripts to separate repository selection, version extraction, and artifact resolution into distinct phases

### Added in 4.0.0

- Semantic version support (`X.Y.Z`) for macOS and Windows â€” resolves the latest build for the given version from the S3 bucket
- Error handling when no matching artifact is found in the S3 bucket
